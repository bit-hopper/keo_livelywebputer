Here's a practical guide to integrating AT Protocol into your app for authentication and data storage:

Ignore Go implementation

---

## 1. Understand the Core Concepts

- **DID (did:plc):** A persistent, globally unique identifier for each user. It resolves to a DID Document containing signing keys and the user's PDS location. [[AT Protocol DIDs](https://atproto.com/specs/did)]
- **PDS (Personal Data Server):** Hosts the user's data repository, manages identity, and acts as the OAuth Authorization Server. [[PDS overview](https://atproto.com/guides/the-at-stack#pds)]
- **Repository:** Each user has a signed, public data repo on their PDS where records (posts, likes, custom data) are stored. [[Repository spec](https://atproto.com/specs/repository)]

---

## 2. Authentication: Use OAuth

AT Protocol uses **OAuth 2.1** as the primary authentication mechanism. You should **not** use the legacy `createSession`/App Password approach for new apps. [[OAuth spec](https://atproto.com/specs/oauth)]

### Key points:
- Your app (OAuth client) does **not** pre-register with any PDS — client discovery is automated via a **client metadata document** you host at a public URL.
- The user provides their handle or DID; your client resolves it to their PDS, which acts as the Authorization Server.
- Only the `atproto` scope is needed for basic "login with atproto" (identity authentication). Add resource scopes for data access. [[Authorization scopes](https://atproto.com/specs/oauth#authorization-scopes)]

### Steps:

**a. Host a client metadata document** at a public HTTPS URL (this is your `client_id`):
```json
{
  "client_id": "https://yourapp.com/oauth-client-metadata.json",
  "client_name": "Your App Name",
  "client_uri": "https://yourapp.com",
  "redirect_uris": ["https://yourapp.com/oauth/callback"],
  "scope": "atproto",
  "token_endpoint_auth_method": "none",
  "response_types": ["code"],
  "grant_types": ["authorization_code", "refresh_token"],
  "dpop_bound_access_tokens": true
}
```

**b. Start the OAuth flow** — ask the user for their handle or DID, resolve it to their PDS, and redirect them to authorize. Use an SDK like `@atproto/oauth-client-node` (Node.js) or the Go SDK:

```typescript
// Node.js example (loopback/dev)
import { NodeOAuthClient, buildAtprotoLoopbackClientMetadata } from "@atproto/oauth-client-node";

const client = new NodeOAuthClient({
  clientMetadata: buildAtprotoLoopbackClientMetadata({
    scope: "atproto",
    redirect_uris: ["http://127.0.0.1:3000/oauth/callback"],
  }),
  stateStore: { /* implement get/set/del */ },
  sessionStore: { /* implement get/set/del */ },
});

// Initiate login with user's handle
const authUrl = await client.authorize("alice.bsky.social", { scope: "atproto" });
// Redirect user to authUrl...

// After callback:
const { session } = await client.callback(callbackParams);
```
[[OAuth tutorial](https://atproto.com/guides/oauth-tutorial#part-2-implementing-o-auth)]

**c. After authorization**, the Authorization Server returns the user's `did` in the `sub` field. This is their canonical identifier — store and use it. [[Identity authentication](https://atproto.com/specs/oauth#identity-authentication)]

---

## 3. Account Creation

For users who don't have an ATProto account yet, you have two main options:

- **Direct users to an existing PDS** (like `bsky.social`) to create an account, then OAuth into your app.
- **Host your own PDS** alongside your app to allow in-app signups. The PDS now supports an OAuth-based account creation flow using `prompt=create`, which keeps the experience within your app's flow. [[Network account management](https://atproto.com/blog/network-account-management)]

> For new-to-atproto users, hosting your own PDS is the smoothest path. The PDS reference implementation includes a web interface for sign-up and account management. [[self-hosting](https://atproto.com/guides/self-hosting)]

---

## 4. Reading and Writing Data

Once authenticated, you can read/write records in the user's PDS repository.

### Writing a record (e.g., a post):
```typescript
import { Client } from "@atproto/lex";
// session = OAuthSession from step 2

const client = new Client(session);

const result = await client.create(app.bsky.feed.post, {
  text: "Hello from my app!",
  createdAt: new Date().toISOString(),
});

console.log(result.uri); // at://did:plc:.../app.bsky.feed.post/...
```
[[Writing data](https://atproto.com/guides/writing-data#writing-data)]

### Reading a record:
```typescript
const post = await client.get(app.bsky.feed.post, { rkey: "3jxf7z2k3q2" });
```

For **custom data**, define your own [Lexicon schemas](https://atproto.com/specs/lexicon) under a namespace you control (e.g., `com.yourapp.data`), and store records under those collections.

---

## 5. Permissions (Scopes)

Request only the permissions your app needs. For example:
- `atproto` — identity/login only
- `repo:app.bsky.feed.post?action=create` — create Bluesky posts
- `repo:com.yourapp.data` — read/write your custom records

[[Permission spec](https://atproto.com/specs/permission)]

---

## Summary

| Step | What to do |
|------|-----------|
| Auth | Implement OAuth with a hosted client metadata document |
| Account creation | Host your own PDS or direct users to an existing one |
| Data storage | Write records to the user's PDS repo using Lexicon-typed collections |
| Identity | Use the `did` (`sub`) returned by OAuth as the canonical user ID |

The [OAuth tutorial](https://atproto.com/guides/oauth-tutorial) and the [Go OAuth example](https://github.com/bluesky-social/cookbook/blob/main/go-oauth-web-app/README.md) in the cookbook are good practical starting points.

### Key Decisions

#### 1. User Identity (Same as AT Protocol)
- Every Lively user gets a DID (Decentralized Identifier)
- Federated users: `did:plc:...` (from AT Protocol)
- Local users: `did:lively:username.example.com` or `did:key:...`
- DID is the canonical user identifier in all systems

#### 2. Account Types
- **local**: Username/email signup on Lively instance
- **atproto**: Existing AT Protocol account (OAuth federated)
- **federated**: Local account but sync-able with other instances

#### 3. Key Management
- Private keys stored in external KMS (AWS KMS, HashiCorp Vault, etc.)
- Database stores only:
  - Key identifiers
  - Public keys (for verification)
  - Key metadata (creation date, algorithm, active status)
- Never store private keys in DB

#### 4. Data Storage (PDS/Repository)
- File structure: `pds/users/{username}/repo/`
- Uses AT Protocol MST (Merkle Search Tree) format
- Records: `com.lively.user.{profile|settings|world|encryption}`
- All data encrypted with JOSE (JWE)
- Commits are cryptographically signed

#### 5. Token Management
- OAuth tokens encrypted in `sessions.oauth_*_token`
- Access tokens: ~1 hour expiry
- Refresh tokens: ~90 day expiry
- Automatic refresh on expiry

#### 6. RBAC (Role-Based Access Control)
- Roles: admin, moderator, user, guest
- Scopes: `repo:com.lively.*`, `repo:app.bsky.*`
- Actions: create, read, update, delete
- Expiry support for temporary permissions

### Performance Considerations

- **Session queries**: Index on (user_id, is_active, expires_at)
- **PDS record lookups**: Index on (user_id, collection, rkey)
- **OAuth state cleanup**: Periodic job to delete expired oauth_state rows
- **MST caching**: Cache current repo_root_cid per user in Redis
- **KMS calls**: Batch operations when possible, cache public keys locally

### Security Audit Points

1. **Token encryption**: Verify AES-256-GCM encryption for OAuth tokens
2. **Key isolation**: Ensure no private keys in DB or app memory
3. **Session security**: Verify IP+user-agent tracking
4. **JOSE validation**: Verify JWE headers match expected algorithms
5. **DID signing**: Verify commits signed with correct key from KMS
6. **Audit logging**: All auth/permission changes logged
