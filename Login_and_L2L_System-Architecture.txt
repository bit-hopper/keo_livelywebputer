L2L Connection Flow (client → server)

Browser loads world
  → lively.net.SessionTracker.start()         [SessionTracker.js client]
  → POST /login                               [gets session cookie]
  → WebSocket to /SessionTracker/connect
  → sends "registerClient" {id, user, worldURL, lastActivity}
  → server stores session in memory           [servers/SessionTracker.js:70]
  → ConnectionIndicator turns green
  → heartbeat every N ms keeps alive
Database Relationship
The SQLite3 (objects.sqlite, 29.9 MB) is not a user auth database — it's a versioned object store for code/resource history:


SELECT distinct(author) FROM versioned_objects
WHERE path = ? AND date > (SELECT max(date) FROM versioned_objects WHERE path = ? AND change = 'deletion')

Key Insight
The login system is very lightweight — it's essentially just a named session. There is no password, no JWT, no OAuth. The username is a string stored in browser localStorage. Authentication/authorization is an optional layer on top via AuthServer.js. The l2l system is primarily about multi-user session awareness (who's in which world, chat between sessions, server-to-server federation) rather than security.
